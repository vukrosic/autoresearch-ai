# Changelog

## 0.6.0 — 2026-05-22

Integrated (features that now fire automatically inside the loop):

- **`compare` auto-runs significance.** When `cmdCompare` has at least 2 runs with a finite metric and the best run differs from the worst, it now calls the existing `runSignificanceTest` helper automatically (5000 permutations, 2000 bootstrap, seed 42) and appends a `significance` block: `delta`, `CI95`, Cohen's d, permutation p-value, and a `SIGNIFICANT` / `not significant` verdict at α=0.05. Skips when both sides are single-point scalars (uninformative) or when the user passes `--skip-significance`. Hands the verdict to the agent without it having to know to call `autoresearch significance`.
- **`propose --mode novel` auto-runs mechanism dedup.** Before writing proposals, `cmdPropose` now normalizes each proposal's `mechanism` string and compares against `usedMechanismsFromEvidence(runs, papers, hypotheses)`. Overlapping mechanisms get a `mechanism_duplicate_of` field on the proposal row, and a one-line warning surfaces the first three dupes to stderr. Doesn't drop — flags. Lets the agent see at-a-glance which "novel" ideas the system already tried.
- **`next-experiment` runbook includes a Resource Sanity section.** When a proposal carries arch hints (`proposal.arch.{params, layers, d_model, ...}`) or the repo profile has a `model` block, the generated runbook now includes a "Resource Sanity" section with a ready-to-paste `autoresearch gpu-fit ...` command (arch args pre-filled) and a `autoresearch sweep-projection --n 1` wall-time check. Catches OOM and "this run will take 4 days" before the agent commits.

- **`preflight` auto-runs gpu-fit.** When `--params 7B` (or `--layers + --d-model`) is passed, or when `repo-profile.json` exposes `model.{params, layers, d_model, d_ff, batch_size, seq_len, dtype}`, `autoresearch preflight` now adds a `vram_fit` check using the `analyzeVram` helper from `bin/researchloop-gpu-fit.js`. Reports "✓ fits solo / needs DP=2/4/8" against `--gpu H100` (or the auto-detected local GPU). `--require-fit` upgrades a non-pass to a hard fail so preflight short-circuits before the OOM-causing run launches.
- **`run` / `baseline` auto-run overfit-watch.** After every run finishes, `executeRun` calls `analyzeOverfit` against the streamed `metrics.jsonl`. When both `train_loss` and `val_loss` were captured, the row gets an `overfit` block (`val_best_step`, `val_best_value`, `diverge_step`, `wasted_fraction`, `still_descending`), and the post-run output prints a one-line summary — `overfit: divergence at step 800; val best @ 500; 37.5% of compute spent after the val min` — without the agent needing to know to call `overfit-watch`.
- **`report --format markdown` includes MFU and Overfit Watch sections.** When any row has enough fields for `computeMfuForRow` to compute MFU (wall_seconds + params + tokens + GPU peak), a top-10 MFU table appears with a median + diagnostic verdict (< 20% likely dataloader-bound, 20–35% try FlashAttention-2/torch.compile, ≥ 45% well-tuned). When any row carries the auto-written `row.overfit` block with real divergence or > 10% wasted compute, an Overfit Watch table summarizes val best step, divergence step, and wasted-compute percentage per run.
- **`review` adds `overfit_clean` and `mfu_reasonable` checks.** Promotion now blocks (unless `--force` or `--skip-review`) on runs that diverged with > 25% of compute spent past the val minimum (likely cherry-picked from an overfit run) or whose MFU < 10% (compute almost certainly leaked through dataloader/kernel stalls rather than a real algorithmic win).

New:

- **`autoresearch judge`.** LLM-as-judge eval harness (offline planner). Three modes — `pairwise` (A vs B, judge picks winner / tie, output compatible with `autoresearch elo`), `scalar` (1–10 scoring along accuracy / helpfulness / clarity), `reference` (gold-answer comparison). Generates ready-to-send judge prompts from a candidates JSONL (`--candidates`), supports field aliases (`response`/`completion`/`output`, `target`/`reference`/`gold`), pre-projects total cost using the same baked-in pricing registry as `api-budget` (Anthropic Opus/Sonnet/Haiku, GPT-5/4o/o3, Gemini Pro/Flash, DeepSeek V3) so the researcher sees the bill before any API call. `--parse outputs.jsonl --prompts judge_prompts.jsonl` reads the user's judge outputs back, strips them through a JSON-or-fenced-JSON parser, and emits a clean parsed JSONL that pipes straight into `autoresearch elo` for ratings. Zero-network — the user runs the API calls themselves. Implementation in `bin/researchloop-judge.js`.
- **`autoresearch grad-noise`.** Gradient noise scale (B_simple) estimator from McCandlish et al. 2018 ("An Empirical Model of Large-Batch Training") — the critical batch size beyond which doubling stops giving you ~2× speedup. Two estimators: (1) exact two-batch via `--norm-small N --batch-small N --norm-big N --batch-big N` (logs the gradient L2 norm computed on microbatches of two sizes at the same step, the McCandlish §A.2 identity), and (2) sample-variance proxy from a JSONL of `{step, norm}` or `--id <run-id>` reading `metrics.jsonl` with `metric=grad_norm`. Pass `--batch N` for a verdict — under-batched / near critical (linear-speedup regime [B_simple/4, B_simple×4]) / over-batched. Pure stdlib. Aliased as `gns` / `gradient-noise`. Implementation in `bin/researchloop-grad-noise.js`.
- **`autoresearch memorization`.** Verbatim training-data leakage check on model outputs (Carlini et al. 2021/2023 attack family). Builds the set of all n-character shingles in the training corpus (default n=50), then for each generated output uses an n-gram filter to find candidates and a length-doubling search to surface the longest verbatim match. Reports per-output longest-match length, the offending snippet, and an aggregate memorization rate at `--report-threshold N` (default = n). Loud warning when > 5% of outputs contain verbatim training substrings. Accepts JSONL outputs (`text`/`output`/`response`/`completion`/`generation` field), plain-text one-per-line, single training file `--train`, or comma-separated `--train-globs`. Distinct from `autoresearch canary` (which checks for eval/train overlap in the dataset itself). Aliased as `memo` / `verbatim-check`. Implementation in `bin/researchloop-memorization.js`.
- **`autoresearch mfu`.** Model FLOPs Utilization for every recorded run, or a single `--id`. MFU = (6 × params × tokens / wall_seconds) / (peak_GPU_TFLOPS × world_size). Per-row autodetection: pulls params from `row.config.params`/`n_params` or derives from `row.config.layers + hidden_size + ffn_dim + vocab_size` (Kaplan 2020 transformer rule), pulls batch×seq from `row.config.batch_size + seq_len`, total tokens from `row.tokens` or `steps × tokens/step`, GPU peak from a baked-in registry covering H200/H100/H100-PCIe/A100-{40,80}/L40S/L40/RTX-6000-Ada/4090/3090/A10G/V100/T4/MI300X/MI250X/TPU-v4/v5e/v5p. Project-wide overrides via `--params`, `--batch`, `--seq`, `--tokens`, `--gpu`, `--tflops`, `--gpus`. Output table sorts by MFU descending. Loud verdicts at < 20% (likely dataloader stall), 20–35% (turn on FlashAttention-2, torch.compile, fused optimizer), and ≥ 45% (excellent). Implementation in `bin/researchloop-mfu.js`.
- **`autoresearch kv-cache`.** Inference VRAM estimator. KV cache bytes per token = 2 × layers × n_kv_heads × head_dim × dtype_bytes — exposes GQA explicitly via `--n-kv-heads` vs `--n-heads` so users don't model attention sharing wrong. Reports total VRAM (weights + KV at given batch × context + ~5% workspace), max batch at the given context that fits, max context at batch=1, and max (batch × context) headroom. Built-in GPU SKU registry (H200/H100/A100/L40S/RTX 4090/3090/A10G/V100/T4/MI300X) via `--gpu`, or `--vram-gb N` for arbitrary hardware. Supports `--kv-dtype fp16|bf16|fp8|int8` and `--weights-dtype` independently (vLLM/TRT-LLM style mixed serving). Suggests GQA conversion when MHA is detected with > 8 heads. Aliased as `kv`. Implementation in `bin/researchloop-kv-cache.js`.
- **`autoresearch lr-finder`.** Analyze a Smith-style LR range test (2015, "Cyclical Learning Rates"; fast.ai). Reads `{lr, loss}` rows from `--file lrtest.jsonl` (also accepts `learning_rate`, `train_loss`) or `--id <run-id>` (reads `runs/<id>/lrtest.jsonl`, falling back to `metrics.jsonl` with both `lr` and `value` fields). EMA-smooths the loss (default α=0.05, fast.ai convention), finds the minimum, detects divergence as the first step right of the minimum where smoothed loss > 4× min, and reports the elbow as the lr just before divergence. Suggests `max_lr = elbow / 10` for one-cycle / cosine schedules and `lr = elbow / 30` for constant-lr training. Renders an ASCII curve with `:` marking the min and `|` marking the elbow. Warns when the steepest slope is too shallow (range was too narrow or model is already well-fit). Aliased as `lrfinder` / `find-lr`. Implementation in `bin/researchloop-lr-finder.js`.
- **`autoresearch overfit-watch <run-id>`.** Finds the train/val divergence point in a run's streamed metrics. Reports val minimum step + value (= optimal early-stop point), train value at that step (= the honest generalization), the first step after the val min where the val−train gap exceeds `--gap-threshold` of |val_best| (default 5%), and "wasted compute" as the fraction of steps after the val min. Notes when val is still descending in the last window — i.e. *more* training is the right call, not less. Override metric names via `--train-metric` / `--val-metric` (default `train_loss` / `val_loss`). Aliased as `overfit`. Implementation in `bin/researchloop-overfit-watch.js`.
- **`autoresearch shard-plan`.** Recommends `(TP, PP, DP)` split for a target model given a cluster. Enumerates every `TP × PP × DP = N_GPUs` candidate where TP/PP ∈ {1,2,4,8,16,32,64} and PP divides `--layers`, computes per-GPU memory using the same Korthikanti activation model as `gpu-fit` (with PP dividing weights and activations, TP further dividing both, ZeRO sharding opt/grad/params across DP), and ranks by: feasibility → larger DP (more compute overlap) → smaller TP (TP needs NVLink/NVSwitch domain) → smaller PP (bubble cost). Built-in GPU registry; `--vram-gb` override. Calls out the "TP must be co-resident on a single node" caveat when TP > DP, and the pipeline-bubble micro-batch requirement when PP > 1. Aliased as `parallelism` / `shard`. Implementation in `bin/researchloop-shard-plan.js`.
- **`autoresearch elo --file wins.jsonl`.** Pairwise model ratings. Fits both online Elo (K=4, configurable) and Bradley-Terry MLE via the Hunter 2004 MM algorithm with Laplace smoothing (so a model with zero wins doesn't drive its rating to -∞). 95% stratified bootstrap CIs on BT ratings (default 200 resamples, seeded). Per-model: games / wins / losses / ties / win rate / Elo / BT rating / BT CI95. Pairwise win-rate matrix when ≤ 12 models. Accepts both `{model_a, model_b, winner}` rows and compact `{model_a, model_b, a_wins, b_wins, ties}` count rows; expands counts into individual matches before fitting. Aliased as `arena`. Implementation in `bin/researchloop-elo.js`.
- **`autoresearch api-budget`.** Projects $ cost for an LLM API evaluation. Built-in pricing registry covering Anthropic (Opus 4.7/4.6/4, Sonnet 4.6/4.5/3.5, Haiku 4.5/3.5), OpenAI (GPT-5, GPT-4o, GPT-4o-mini, o1/o1-mini, o3/o3-mini), Google (Gemini 1.5 Pro/Flash, 2.0 Flash, 2.5 Pro), DeepSeek (V3, R1), and open models on Together (Llama 3.1 70B/405B, Qwen 2.5 72B). Two input modes: explicit `--prompts N --avg-input-tokens T --avg-output-tokens U`, or `--file prompts.jsonl` with auto-token-count via the ~4 chars/token rule (tunable via `--tokens-per-char`). Cross-checks the projection against `.researchloop/budget.json` if present, with an "exceeds budget by $X" warning. Always emits cost-reduction tips (cheaper provider, capped output, prompt caching for repeated prefixes). `--list` prints the pricing table; `--input-price` / `--output-price` override registry. Aliased as `api-cost` / `tokens-cost`. Implementation in `bin/researchloop-api-budget.js`.
- **`autoresearch gpu-fit`.** Pre-flight VRAM estimator for transformer training. Per-component breakdown (weights / gradients / optimizer state / activations) per GPU, then a verdict table for H200/H100/A100/L40S/RTX 4090/3090/A10G/V100/T4 at solo / DP=2/4/8. Accepts `--params 7B` for a quick rough number, or `--layers N --d-model N --d-ff N --vocab N` for an arch-derived parameter count plus real activation memory. Handles `--dtype bf16|fp16|fp32|fp8`, `--optimizer adamw|sgd|adam8bit`, `--grad-checkpoint`, `--tp N` (Megatron tensor-parallel sharding), `--dp N` and `--zero 0|1|2|3` (DeepSpeed/FSDP-style sharding of opt/grad/params across data-parallel ranks). Activation memory follows the Korthikanti et al. 2022 ~12-floats-per-token-per-layer rule plus the attention-prob batch×heads×seq×seq term, with selective recomputation divider when `--grad-checkpoint` is on. Aliased as `vram-fit` / `vram`. Implementation in `bin/researchloop-gpu-fit.js`.
- **`autoresearch eval-diff --a runA/predictions.jsonl --b runB/predictions.jsonl`.** Per-example flip analysis between two predictions files. Reports both-correct / both-wrong / A-only / B-only counts, churn rate (% of examples that changed in either direction), accuracy A vs B with net Δ, and lists the top-K regressions and gains by id. Per-class flip table when `target`/`label`/`class` is present. Correctness is read from (in priority) an explicit `--field correct` boolean, a numeric `score` thresholded at 0.5, or `prediction == target` equality. Matches across files by `--id-key id` when both sides have it, falls back to row position with a loud warning. Emits a "metric is the same but the model behaves differently" heads-up when churn > 5% and net delta is near zero — the failure mode every LLM eval has. Aliased as `predictions-diff`. Implementation in `bin/researchloop-eval-diff.js`.
- **`autoresearch scaling-fit`.** Fits a Kaplan/Chinchilla-style power law on the local ledger: `log(loss) = log(c) − α · log(x)`. Auto-detects the independent variable from `row.compute`, `row.flops`, `row.params`, `row.tokens`, or `row.wall_seconds` (override via `--x KEY`). Reports α, prefactor `c`, R², and a sorted (x, y, predicted) table. `--with-offset` switches to the Hoffmann form `y ≈ a + b · x^(-α)` via grid-search on the irreducible-loss offset `a` and linear regression in log-log at each grid point. `--target N` extrapolates to a future budget and reports projected loss + % reduction from current best. Warns when R² < 0.7 (likely mixed architectures or undertrained rows) and when α is outside the typical 0.05–0.35 transformer-LM band. Aliased as `scaling`. Implementation in `bin/researchloop-scaling-fit.js`.
- **`autoresearch sample-efficiency <run-id>`.** Reads the streamed `metrics.jsonl` for one run and reports the step at which the metric first reached 50 / 75 / 90 / 95 / 99% of the (start → final) improvement, plus the fraction of total training that took. Also detects the first plateau (last 10% of the run moves the value by < `--plateau-threshold`, default 0.5% of |value|) and reports % of training spent on the plateau. `--vs <baseline-run-id>` adds a head-to-head: steps each run needed to reach baseline's 50/90/99% milestones, expressed as a speedup ratio. Direction is inferred from goal.md or set via `--direction lower|higher`. Aliased as `se`. Implementation in `bin/researchloop-sample-efficiency.js`.
- **`autoresearch rl-stats <run-id>`.** Robust episode-return aggregator following Agarwal et al. 2021 ("Deep RL at the Edge of the Statistical Precipice", NeurIPS). Reports mean ± std, median, IQM (interquartile mean over the middle 50%, less seed-noise than mean, less censoring than median), p25/p75, min/max, and 95% stratified bootstrap CIs on IQM and median (default 1000 resamples, seeded for reproducibility via a Mulberry32 PRNG). `--success-threshold N` adds a success-rate row from episode_return ≥ N, or honors a per-row boolean `success` field. `--vs <baseline-id>` reports sample efficiency: steps this run needed to reach baseline's IQM, as a speedup ratio. Input formats: `rewards.jsonl` (`{step, episode_return}`, also accepts `reward`, `return`, `score`, `value`) or fallback to `metrics.jsonl` filtered to `metric == episode_return`. `--file PATH` reads any JSONL outside the ledger. Warns when n < 30 episodes and when CI width > 30% of |IQM| — the noise floor where RL claims stop being honest. Aliased as `rl`. Implementation in `bin/researchloop-rl-stats.js`.
- **`autoresearch sweep-projection`.** Pre-flight cost and wall-time projection for a sweep — answers "is this going to take three days?" before launching. Three input modes: `--name <sweep>` reads an already-generated `.researchloop/sweeps/<name>.queue.jsonl` and counts unfinished rows; `--spec FILE.yaml` reads a spec and uses its `budget`; `--n N` for ad-hoc projection. Time-per-run is estimated from (in priority): `--seconds-per-run` override, the median wall_seconds across the top-10 ledger rows whose command shares Jaccard ≥ 0.3 with the sweep's base_command, or the median across the entire ledger. Cost-per-run comes from `.researchloop/cost.yaml` (`hourly_usd`) × seconds/3600, or `--cost-per-run` override. Reports both median and p95 cost so you see the worst-case scenario. Warns at > $100 and > 1 day projected wall. Aliased as `sweep-project` / `project-sweep`. Implementation in `bin/researchloop-sweep-projection.js`.
- **`autoresearch headroom`.** Gap-to-perfect / gap-to-SoTA reality check. Computes (baseline value → current best) as a percentage of the theoretical ceiling gap (default: 0 for loss/error/ppl/mse/rmse/mae/cost/latency metrics, 1 for accuracy/F1/AUC/ROUGE/BLEU/pass-rate/etc.), so a "20% loss reduction" can be reported as "you've closed 36% of the available headroom." `--sota N` adds a public state-of-the-art comparison: gap to SoTA, % of baseline→SoTA closed, and a callout when you're past SoTA. Pulls a seed-variance σ either from a multi-seed aggregate row (`row.seeds.values`) or from same-command run groups (n ≥ 2) and reports your improvement in σ-units — emits a "re-run with more seeds; `autoresearch power --detect-delta` will size N" heads-up when the gain is under 2σ. Metric / direction / baseline / current all overridable; baseline auto-picks from `tags: [baseline]` then root row. Implementation in `bin/researchloop-headroom.js`.
- **`autoresearch significance <run-a> <run-b>`.** Permutation test + bootstrap 95% CI on the metric delta between two runs. Multi-seed aggregate rows (with `seeds.values`) contribute every seed; scalar rows contribute one point. Reports `delta`, `ci_95`, Cohen's d, two-sided permutation p-value, and a verdict at `--alpha 0.05`. `--direction lower|higher` adds a "which side wins" interpretation; `--require-significant` makes the command exit non-zero when `p >= alpha` (use in promotion gates / CI). `--format json` for machine consumption; `--seed N` makes the PRNG deterministic. Aliases as `sig`. Warns loudly when both sides are single-point runs (permutation p is uninformative at n=1 vs n=1) and when min n < 3 (CI and effect size will be wide). Pure stdlib — no scipy, no extra deps.
- **`autoresearch eval`.** Structured eval runner for recorded runs. Reads `.researchloop/eval.yaml`, runs the configured `eval_command`, parses every declared metric from stdout or a file, writes them back onto the matching row in `runs.jsonl`, and preserves `null` plus `parse_warnings` when a metric regex does not match. `autoresearch run` and `autoresearch baseline` call it automatically when the eval config is present.
- **`autoresearch propose`.** Writes grounded experiment proposals to `.researchloop/scratchpad/proposals.jsonl` by reading the goal/baseline, repo profile, paper notes, hypothesis notes, and the run ledger. Proposals are content-hashed, id-stable, and point at real file paths when the repo profile or file walk can resolve them. `--mode novel` keeps only mechanism-first ideas with kill criteria instead of sweep-shaped ideas.
- **`autoresearch propose --with-priors`.** Proposal generation can now attach arXiv prior-art evidence in the same pass, write missing paper notes under `.researchloop/scratchpad/papers/`, and update existing proposal rows without changing their content-stable ids. Use `--prior-limit`, `--offline`, and `--cache-dir` for controlled runs.
- **`autoresearch rank`.** Scores proposal rows with explainable `impact`, `cost`, `risk`, `novelty_vs_runs`, and `evidence` values, then writes `.researchloop/scratchpad/ranked-proposals.jsonl` and `ranked-proposals.md` when `--write` is set. The score is deterministic across invocations and a duplicate of the goal's best run gets very low novelty.
- **`autoresearch next-experiment`.** Turns the top ranked proposal, or an explicit `--proposal`, into an agent-ready experiment runbook under `.researchloop/scratchpad/experiments/`. The runbook carries the hypothesis, mechanism, edit targets, prior evidence, preflight/smoke/run commands, kill criterion, and follow-up compare/story/promote commands. `--script FILE.sh` emits a reviewable shell wrapper for the execution commands.
- **`autoresearch topic`.** Topic intake now surfaces the nearest paper notes and run rows from the local scratchpad, ranks them against the topic text, and prints the exact `paper-read` / `hypothesis` follow-up commands for the most relevant evidence.
- **`autoresearch paper-read`.** Turns a paper note or arXiv paper into a structured baseline-aware note with `claim`, `mechanism`, `limits`, `how to port this`, and `baseline relevance`. Supports local notes, exact arXiv fetch by id, and cache-backed offline re-reads.
- **`autoresearch priors`.** Prior-art lookup now attaches a deduped `priors` array to one proposal row, writes any missing paper notes under `.researchloop/scratchpad/papers/`, and reuses the arXiv fixture/cache path so offline runs still work when cached XML exists.
- **`autoresearch hypothesis`.** Writes mechanism-first hypotheses from paper notes or recorded runs. `--from-papers`, `--from-runs`, `--paper-id`, `--run-id`, and `--novel` all work, and `--novel` rejects sweep-like mechanisms instead of pretending a parameter search is a hypothesis.
- **Run lineage dashboard view (G17).** `/lineage` and `/api/lineage` now render the parent-child run chain created by `replay`, `resume`, and `verify`. The dashboard panel shows roots, descendants, and parent ids, and `test:dashboard` now checks the lineage route and API.
- **`autoresearch search "TEXT"`.** Full-text grep across the project's research artifacts: paper notes, hypothesis notes, proposals, learnings, archive markers, winners, goal, plan, and (optionally) every ledger row. Case-insensitive, substring match, ranked by file mtime, shows a 60-char context window around each hit. `--include ledger,papers,...` narrows the scope; `--context N` widens or tightens the snippet. Aliased as `grep`. Implementation in `bin/researchloop-search.js`.
- **`autoresearch tail <run-id>`.** Wraps the OS `tail` so the user doesn't have to remember `.researchloop/scratchpad/runs/<id>/log.txt`. `--follow`/`-f` streams new lines; `--lines N` controls the initial window. `--metrics` switches sources to the run's `metrics.jsonl` and pretty-prints each streamed step+value sample (with `--follow`, polls every 500ms). `--latest` auto-picks the most-recent ledger row when no id is given. Implementation in `bin/researchloop-tail.js`.
- **`autoresearch container-snapshot --id <run-id>`.** Writes a best-effort reproduction kit under `.researchloop/scratchpad/runs/<id>/repro/`: a Dockerfile (CPU `python:X.Y-slim` by default, or `nvidia/cuda:...` with `--gpu`), `requirements.lock` from the run's captured `env.pip_freeze` (when available), and a README explaining what's pinned, what isn't, and how to build + run. Doesn't guarantee bitwise reproducibility — that needs `autoresearch determinism` on the same hardware — but gets a reviewer ~95% of the way there. Aliased as `container`. Implementation in `bin/researchloop-container.js`.
- **`autoresearch warmstart <run-id>`.** Generates a launch-command snippet that starts a new experiment from an existing run's last checkpoint. Distinct from `resume` (which continues the SAME run): warmstart says "use that run's final weights as starting weights for a NEW experiment with possibly different hyperparameters." Reads `last_checkpoint` from the parent row (override via `--checkpoint PATH`), splices in the `resume_flag_template` from `eval.yaml` (default `--resume {path}`), generates a fresh id, and prints (or `--out FILE.sh` writes) a ready-to-modify `autoresearch run --command "..."` block. Implementation in `bin/researchloop-warmstart.js`.
- **`autoresearch gpu-report`.** Aggregates per-run `system.jsonl` samples (G32, every 5s by default) into a load + memory pressure table: per-run mean/p95/max load (`os.loadavg()[0]`), mean/p95/max memory used fraction, plus across-ledger totals (total wall hours, longest run, heaviest load, heaviest memory). True per-GPU utilization will plug into the same pipeline as soon as an `nvidia-smi` sampler ships. Aliased as `gpu`. Implementation in `bin/researchloop-gpu-report.js`.
- **`autoresearch hooks`.** Management surface for pre/post/promote shell hooks living at `.researchloop/hooks/<event>.sh` (or `<event>.d/*.sh` for multiple). Events: `pre_run`, `post_run`, `on_promote`, `on_failure`, `on_archive`. Subcommands: `list` (default), `install <event>` (drops a documented stub), `remove <event>` (or `--file PATH` for `.d/` members), `test <event>` (sources every matching script with realistic env vars and reports exit codes), `env-template` (prints every `RESEARCHLOOP_*` env var hooks can rely on). Aliased as `hook`. Implementation in `bin/researchloop-hooks.js`.
- **`autoresearch data-sample --path FILE`.** Quick look at a data file: autodetects JSONL/CSV/TSV/text, reservoir-samples N rows (seeded for reproducibility), reports length percentiles (p1/p50/p95/p99/min/max), and — for JSONL — a field-presence histogram across every row plus a `--label FIELD` class-balance distribution. Loud warning when one class is > 90% of the data. Aliased as `sample-data`. Implementation in `bin/researchloop-data-sample.js`.
- **`autoresearch canary --eval EVAL.jsonl --train TRAIN.jsonl`.** Data-leak detector. Hashes every row (sha256 of normalized content; JSONL is JSON-sorted so insignificant whitespace/key-order doesn't matter) on both sides and reports exact overlap. `--substring` switches to substring containment (catches paraphrase-light leakage), `--min-len N` filters out trivially-short rows. Verdict: `clean` / `some_contamination` / `heavy_contamination` (≥5% overlap). Exits 2 on any overlap so CI can block contaminated eval runs. Implementation in `bin/researchloop-canary.js`.
- **`autoresearch stale-locks`.** Finds and (with `--clean`) removes zombie lock files left behind when a sweep worker dies (SIGKILL, OOM, reboot). Walks `.researchloop/*.lock/` and `.researchloop/tasks.lock/`, reads each lock's pid (json or one-line int), and tags it stale if the pid no longer exists OR (when no pid) if the file is older than `--max-age` minutes (default 60). Without `--clean`, just lists. Aliased as `locks`. Implementation in `bin/researchloop-stale-locks.js`.

## 0.4.0 — 2026-05-20

New commands:

- `autoresearch run --seeds N` — runs the command N times under different seeds (substituted as `{seed}` in the command, also exported as `$RESEARCHLOOP_SEED`). Records one child run per seed plus an aggregator row carrying `mean`, `std`, `min`, `max`, and `child_run_ids` in the ledger.
- `autoresearch loop --command CMD [--iters N]` — closed ratchet loop: runs the command N times, tracks the running-best metric in `.researchloop/scratchpad/loop_state.json`. Optional `--patch-cmd CMD`, `--revert-on-regression`, `--commit-on-win`, and `--keep-if better|same`.
- `autoresearch anomalies [--id RUN_ID] [--format text|json]` — scans recorded `metric_history` for divergence (NaN/inf), spikes (>5× median prior), and plateaus (last 8 steps within 0.5% range).

Improved:

- Every `run` and `baseline` now records GPU stats when `nvidia-smi` is available: `gpu_present`, `gpu_count`, `gpu_util_max_pct`, `gpu_util_mean_pct`, `gpu_memory_peak_mb`, `gpu_memory_total_mb`, and `gpu_hours`. The fields are present (and null) on non-GPU hosts so the ledger schema stays stable.
- `compare` now emits `gpu_runs`, `gpu_hours_total`, and `gpu_memory_peak_mb` summary lines when any compared run has GPU stats.
- `cmdRun` body is now factored through an `executeRun(opts)` helper used by `run`, `baseline`, `sweep`, `loop`, and `--seeds` — no behavioral change for the existing two commands.

Tests:

- New: `test:sweep`, `test:seeds`, `test:anomalies`, `test:loop`, `test:gpu-ledger`, all wired into `npm test`.

## 0.2.0

New:
